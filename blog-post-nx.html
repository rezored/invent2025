Why we chose Nx for Inviteathon's Monorepo?

When we started Inviteathon, we knew it would grow beyond a single Angular app. We had an admin portal, a customer facing dashboard, and a set of shared UI libraries.

The Problem with Multi-Repo

Previously, we maintained separate repositories for each project. Sharing code meant publishing private NPM packages or using git submodules—both of which introduced friction.

Enter Nx

Nx allowed us to keep everything in a single repo while maintaining strict boundaries via tags in nx.json. 

{
  "projects": {
    "inviteathon-client": { "tags": ["type:app", "scope:client"] },
    "inviteathon-admin": { "tags": ["type:app", "scope:admin"] },
    "shared-ui": { "tags": ["type:lib", "scope:shared"] }
  }
}

Now, if we try to import an admin-only library into the client app, the linter stops us immediately. CI/CD only runs for affected projects.
